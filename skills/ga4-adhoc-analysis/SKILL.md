# GA4アドホック分析の壁打ち

## ワークフロー

```
1. ユーザーの分析したい内容をヒアリング
   ↓
2. 適切なdimensions/metricsの組み合わせを提案
   ↓
3. レポート実行・結果の解釈を提示
   ↓
4. 追加の深掘り軸を提案 → 繰り返し
```

---

## Step 1: ヒアリング

`AskUserQuestion` で以下を確認する:

- **分析の目的:** 何を知りたいか（例: 「モバイルユーザーの離脱が多い原因」）
- **property_id:** 環境変数 `GA4_PROPERTY_ID` から取得。未設定の場合はユーザーに確認する
- **期間:** デフォルト 直近7日間

> ユーザーの回答が曖昧な場合は、以下の分析テンプレートを提示して選択してもらう。

### 分析テンプレート

| # | テーマ | 分析内容 |
|---|---|---|
| 1 | ユーザー行動分析 | どのページが使われていて、どこで離脱するか |
| 2 | デバイス分析 | デバイス間でUXに差があるか |
| 3 | 流入分析 | どこからユーザーが来て、何をするか |
| 4 | 機能利用分析 | 各機能の利用状況と定着度 |
| 5 | 時間帯分析 | いつ使われているか、ピーク時の傾向 |
| 6 | コンバージョン分析 | 目標達成までのファネル |
| 7 | 自由入力 | 上記以外のカスタム分析 |

---

## Step 2: クエリ設計

ユーザーの分析目的に基づいて、適切なdimensions/metricsの組み合わせを設計する。

### 利用可能なディメンション（主要なもの）

| カテゴリ | ディメンション | 用途 |
|---|---|---|
| ページ | `pagePath`, `pageTitle`, `landingPage` | ページ別分析 |
| ユーザー属性 | `deviceCategory`, `operatingSystem`, `browser` | デバイス・環境分析 |
| 流入 | `sessionDefaultChannelGroup`, `sessionSource`, `sessionMedium` | 流入経路分析 |
| 時間 | `date`, `hour`, `dayOfWeek` | 時間帯分析 |
| イベント | `eventName` | イベント分析 |
| 地域 | `country`, `city`, `region` | 地域分析 |

### 利用可能なメトリクス（主要なもの）

| カテゴリ | メトリクス | 用途 |
|---|---|---|
| 基本 | `activeUsers`, `sessions`, `screenPageViews` | 規模の把握 |
| エンゲージメント | `engagementRate`, `averageSessionDuration`, `bounceRate` | 質の評価 |
| イベント | `eventCount`, `conversions` | 行動の計測 |
| 収益 | `totalRevenue` | 事業成果 |

### クエリ設計の提示

設計したクエリをユーザーに提示する:

```markdown
## クエリ設計

**目的:** [ユーザーの分析目的]

**取得するレポート:**

### レポート1: [レポート名]
- **ディメンション:** [dimensions]
- **メトリクス:** [metrics]
- **フィルタ:** [あれば]
- **並べ替え:** [order_bys]
- **期間:** [date_ranges]

### レポート2: [レポート名]
（必要に応じて複数）

このクエリで実行してよいですか？
```

---

## Step 3: レポート実行

`mcp__analytics-mcp__run_report` で設計したクエリを実行する。複数レポートは**並列取得**する。

---

## Step 4: 結果の解釈

```markdown
# アドホック分析レポート
目的: [分析目的]
対象: [プロパティ名]（property_id: [ID]）
期間: YYYY/MM/DD〜YYYY/MM/DD

## レポート1: [レポート名]

[テーブル形式でデータ出力]

### 読み取れること
- [データから読み取れるポイント1]
- [データから読み取れるポイント2]
- [データから読み取れるポイント3]

## レポート2: [レポート名]
（同構成）

## 分析まとめ
- **発見事項:** [主要な発見を箇条書き]
- **仮説:** [データから立てられる仮説]
- **注意点:** [データの限界やバイアス]
```

---

## Step 5: 深掘り提案

結果に基づいて、追加の分析軸を提案する:

```markdown
## 深掘り提案

結果を踏まえて、以下の追加分析が有効です:

1. **[分析名]:** [理由]（例: ページ×デバイスのクロス分析で、モバイルの課題を特定）
2. **[分析名]:** [理由]
3. **[分析名]:** [理由]

追加で分析しますか？
```

`AskUserQuestion` で確認し、ユーザーが希望すれば Step 2 に戻って繰り返す。

---

## 出力フォーマット規約

- 数値はカンマ区切り（例: `1,234`）
- パーセンテージは小数点1桁（例: `89.8%`）
- 時間は「○分○秒」形式（例: `5分12秒`）
- 前期比は `(前期比: +12.3%)` or `(前期比: -8.5%)` 形式
- 根拠データは全て GA4 API レスポンスから直接引用。推測値には「推定」と明記
- テーブルの数値は右寄せ（`---:`）

## 注意事項

- GA4 Data APIにはサンプリング・クォータ制限がある。レスポンスの `sampling_metadatas` を確認し、サンプリングされている場合はレポートに明記すること
- セグメント機能は未対応。`dimension_filter` で代替する
- 1レポートのディメンション上限は9個。必要に応じて複数レポートに分割する
- ハルシネーション防止: 数値は必ずAPIレスポンスから直接引用し、計算値は計算過程を示す
